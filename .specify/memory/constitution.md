# Docusaurus Book Documentation Constitution

## Core Principles

### I. Docusaurus-First Architecture

**The entire UI, routing, theming, sidebar, and layout system MUST be handled by Docusaurus, with Tailwind CSS for utility-first styling and shadcn/ui for reusable components.**
No custom routing layers, no competing primary frameworks. Docusaurus is the single source of truth for:
- Navigation and routing (`@docusaurus/router`)
- Theming and styling (Tailwind CSS utilities + Infima CSS framework + custom CSS)
- Sidebar generation and configuration
- Layout system (docs plugin + custom layouts)
- Search integration (Algolia or built-in)
- Plugin ecosystem for all enhancements

**Rationale**: Leveraging Docusaurus fully ensures performance, maintainability, and compatibility
with the ecosystem. Integrating Tailwind CSS and shadcn/ui provides a modern, highly customizable,
and efficient styling and component development workflow within the Docusaurus architecture.
Custom implementations of core Docusaurus functionalities create maintenance burden and break upgrade paths.

### II. Book-Like Content Organization

**Content MUST follow a hierarchical module-chapter-section structure that mirrors a physical book.**

Organization rules:
- **Modules** are top-level books (e.g., "The Robotic Nervous System")
- **Chapters** are major divisions within modules (e.g., "ROS Middleware", "Nodes and Topics")
- **Sections** are subsections within chapters (optional, for complex topics)
- All content MUST be written in MDX (Markdown + JSX)
- File structure MUST be flat within each module for autogenerated sidebars
- Naming convention: `##-chapter-name.mdx` (number prefix for ordering)

Directory structure:
```
docs/
├── module-01-robotic-nervous-system/
│   ├── 01-ros-middleware.mdx
│   ├── 02-nodes-topics-services.mdx
│   ├── 03-python-rclpy-bridge.mdx
│   └── 04-urdf-humanoids.mdx
├── module-02-[next-module]/
│   └── ...
└── intro.mdx
```

**Rationale**: This structure is scalable, version-control friendly, and works seamlessly with
Docusaurus autogenerated sidebars. The flat structure within modules prevents deep nesting
that breaks mobile navigation.

### III. Mobile-First, Reading-Optimized Design

**The reading experience MUST prioritize mobile devices and long-form content consumption.**

Design mandates:
- Mobile drawer navigation with persistent sidebar on desktop
- Minimum 16px body font size, 1.6-1.8 line-height for readability
- Maximum 75ch (characters) content width for optimal reading
- Dark/light mode support (Docusaurus built-in)
- Print-friendly styles (hide navigation, optimize spacing)
- Reading progress indicator for long chapters
- Scroll restoration when navigating between chapters

**Rationale**: Documentation is consumed on all devices. Poor mobile experience leads to
abandonment. Reading-optimized typography reduces cognitive load and improves comprehension.

### IV. Performance and Progressive Enhancement

**The site MUST load fast and remain functional without JavaScript.**

Performance requirements:
- Initial page load: < 3 seconds on 3G
- Time to Interactive (TTI): < 5 seconds on mobile
- Lighthouse Performance score: ≥ 90
- Core Web Vitals: All "Good" ratings
- No layout shift during page load (CLS < 0.1)
- Static HTML generation for all content (Docusaurus SSG)
- Code splitting by route (automatic with Docusaurus)
- Lazy loading for images and heavy components

Progressive enhancement:
- Navigation works without JavaScript (static HTML links)
- Search degrades gracefully (show message if unavailable)
- Interactive components enhance but aren't required for reading

**Rationale**: Fast sites improve user retention and SEO. Static generation ensures content is
accessible even if JavaScript fails. Users on slow connections must still access content.

### V. Component Reusability and MDX Integration

**Reusable React components MUST be available within MDX for rich content without HTML pollution.**

Component requirements:
- All components in `src/components/` directory
- Export via `src/theme/MDXComponents.js` for global MDX availability
- Components MUST be accessible (WCAG 2.1 AA minimum)
- Components MUST work with dark/light themes
- Components MUST be documented with prop types and examples

Standard component library:
- `<Callout type="info|warning|danger|success">` - Highlighted boxes
- `<CodeBlock language="python">` - Enhanced code blocks (built-in)
- `<Tabs>` and `<TabItem>` - Content tabs (built-in)
- `<ReadingProgress />` - Chapter progress indicator
- `<Mermaid>` - Diagram support
- `<VideoEmbed>` - Responsive video embedding
- `<ExerciseBox>` - Interactive exercises (future)

**Rationale**: MDX enables rich content creation without breaking out of Markdown. Reusable
components ensure consistency and reduce duplication. Accessibility is non-negotiable.

### VI. SEO and Discoverability

**Every page MUST have proper metadata for search engines and social sharing.**

SEO requirements:
- Unique `<title>` for every page (via frontmatter)
- Meta description (155-160 characters, via frontmatter)
- Open Graph tags for social sharing (image, title, description)
- Canonical URLs to prevent duplicate content
- Structured data (JSON-LD) for documentation pages
- XML sitemap generation (Docusaurus plugin)
- robots.txt configuration

Frontmatter template:
```yaml
---
id: unique-slug
title: "Chapter Title - Module Name"
description: "Concise chapter summary for search engines"
sidebar_label: "Short Title"
keywords: [ros2, robotics, python, rclpy]
image: /img/social-card-chapter.png
---
```

**Rationale**: Documentation must be discoverable through search engines. Proper metadata
improves click-through rates from search results and social media shares.

### VII. Versioning and Future-Proofing

**The content structure MUST support versioning without major refactoring.**

Versioning strategy:
- Use Docusaurus versioning plugin (when needed)
- Version entire documentation set, not individual modules
- Versions stored in `versioned_docs/` and `versioned_sidebars/`
- Current (unreleased) content always in `docs/`
- Version dropdown in navbar when multiple versions exist
- No breaking changes to module structure between versions

Version numbering:
- Semantic versioning: MAJOR.MINOR (e.g., 1.0, 1.1, 2.0)
- Major version: Complete content rewrites or restructuring
- Minor version: New modules added or significant chapter additions

**Rationale**: Technical content evolves. Versioning allows maintaining old documentation
while publishing new content. The structure must accommodate this from day one.

### VIII. Analytics and Continuous Improvement

**User behavior MUST be tracked to identify content gaps and improve UX.**

Analytics requirements:
- Google Analytics 4 or Plausible (privacy-friendly alternative)
- Track: page views, time on page, scroll depth, search queries
- Custom events: code copy button clicks, external link clicks
- No tracking of PII (Personally Identifiable Information)
- Cookie consent banner (if using GA4 in EU)
- Privacy policy page

Metrics for improvement:
- High bounce rate chapters → need better intros or navigation
- Low scroll depth → content too long or not engaging
- Popular search queries with no results → missing content
- Most visited pages → prioritize updates for these

**Rationale**: Data-driven decisions improve documentation quality. Tracking user behavior
reveals what content is valuable and what needs improvement.

## Content Standards

### Writing Style

- **Voice**: Second person ("you"), active voice, present tense
- **Tone**: Professional but approachable, avoid jargon without definition
- **Structure**: Introduction → Concepts → Examples → Practice
- **Code examples**: Complete, runnable, with expected output
- **Explanations**: "Why" before "How", concepts before commands

### MDX Authoring Rules

- **Frontmatter**: Required for all chapter files (see Principle VI)
- **Headings**: Only one H1 per file (title), start content with H2
- **Code blocks**: Always specify language for syntax highlighting
- **Links**: Relative links for internal pages, absolute for external
- **Images**: Store in `static/img/`, use descriptive alt text
- **Tables**: Markdown tables only (no HTML `<table>`)
- **Admonitions**: Use `:::note`, `:::tip`, `:::warning`, `:::danger` (Docusaurus built-in)

### Code Quality

- **Code examples**: Tested and verified to work
- **Formatting**: Auto-formatted with Prettier (configured for MDX)
- **Linting**: ESLint for React components, Markdownlint for MDX
- **Comments**: Explain non-obvious logic in code examples
- **Imports**: Show all necessary imports in examples

### Component Implementation

**Example: Reading Progress Component**

```javascript
// src/components/ReadingProgress.js
import React, { useState, useEffect } from 'react';
import styles from './ReadingProgress.module.css';

export default function ReadingProgress() {
  const [progress, setProgress] = useState(0);

  useEffect(() => {
    const updateProgress = () => {
      const scrollTop = window.scrollY;
      const docHeight = document.documentElement.scrollHeight - window.innerHeight;
      const scrollPercent = (scrollTop / docHeight) * 100;
      setProgress(scrollPercent);
    };

    window.addEventListener('scroll', updateProgress);
    return () => window.removeEventListener('scroll', updateProgress);
  }, []);

  return (
    <div className={styles.progressBar} style={{ width: `${progress}%` }} />
  );
}
```

## Development Workflow

### Initial Setup

```bash
# Install Docusaurus
npx create-docusaurus@latest docusaurus-book classic --typescript

# Install additional plugins
npm install --save @docusaurus/plugin-pwa
npm install --save docusaurus-plugin-image-zoom
npm install --save @easyops-cn/docusaurus-search-local

# Development server
npm start

# Production build
npm run build

# Serve production build locally
npm run serve
```

### Content Creation Workflow

1. **Create new chapter**: Add `##-chapter-name.mdx` in module directory
2. **Add frontmatter**: Include all required metadata (title, description, keywords)
3. **Write content**: Follow writing style guidelines
4. **Add to sidebar**: Update `sidebars.js` or use autogenerated
5. **Test locally**: Run `npm start` and verify rendering
6. **Validate links**: Check all internal/external links work
7. **Review accessibility**: Test with keyboard navigation and screen reader
8. **Commit**: Use conventional commits (e.g., `docs: add URDF chapter`)

### Testing and Quality Assurance

**Pre-deployment checklist**:
- [ ] All pages render without errors
- [ ] All links are valid (internal and external)
- [ ] Images load correctly with alt text
- [ ] Code blocks have language specified
- [ ] Mobile navigation works on small screens
- [ ] Dark/light mode switching works
- [ ] Search returns relevant results
- [ ] Lighthouse score ≥ 90 for performance
- [ ] No console errors or warnings
- [ ] Print preview shows clean output

**Automated checks** (CI/CD):
```bash
# Build check
npm run build

# Link checking
npx broken-link-checker http://localhost:3000

# Lighthouse CI
npm install -g @lhci/cli
lhci autorun
```

## Deployment and Hosting

### Static Hosting Options

**Recommended platforms**:
- **Vercel**: Zero-config, automatic deployments from Git
- **Netlify**: Similar to Vercel, great build plugins
- **GitHub Pages**: Free for public repos, requires workflow setup
- **AWS S3 + CloudFront**: More control, requires configuration

**Deployment configuration** (example for Vercel):
```json
{
  "buildCommand": "npm run build",
  "outputDirectory": "build",
  "framework": "docusaurus",
  "installCommand": "npm install"
}
```

### Environment Variables

Required for production:
```
SITE_URL=https://docs.example.com
GOOGLE_ANALYTICS_ID=G-XXXXXXXXXX
ALGOLIA_APP_ID=your_app_id
ALGOLIA_API_KEY=your_api_key
ALGOLIA_INDEX_NAME=your_index
```

### CDN and Performance

**Caching strategy**:
- HTML: `Cache-Control: public, max-age=0, must-revalidate`
- CSS/JS: `Cache-Control: public, max-age=31536000, immutable`
- Images: `Cache-Control: public, max-age=31536000, immutable`

**Asset optimization**:
- Enable Brotli/Gzip compression
- Use WebP format for images (with fallback)
- Lazy load images below the fold
- Preload critical fonts

## Governance

### Constitution Authority

This constitution supersedes all other documentation practices for this project. Any
deviation must be documented as an exception with clear justification.

### Amendment Process

1. **Proposal**: Document the proposed change with rationale
2. **Review**: Discuss impact on existing content and architecture
3. **Approval**: Requires project lead sign-off
4. **Migration plan**: Document steps to bring existing content into compliance
5. **Version bump**: Increment constitution version per semantic versioning
6. **Communication**: Notify all contributors of the change

### Version Numbering

- **MAJOR** (X.0.0): Backward-incompatible changes (e.g., switch from Docusaurus to another platform)
- **MINOR** (1.X.0): New principles or sections added, material expansions
- **PATCH** (1.0.X): Clarifications, typo fixes, non-semantic refinements

### Compliance

- All pull requests MUST be reviewed against this constitution
- Automated checks MUST enforce linting, formatting, and link validation
- Non-compliance MUST be flagged in code review
- Complexity additions MUST be justified against Principle IV (Performance)
